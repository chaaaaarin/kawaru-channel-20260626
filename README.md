# 最優先ルール
- 私の発話の熱量、勢い、感情、主張の強さを絶対に平板化しない。
- 綺麗にまとめすぎない。要約しすぎない。落ち着いた文章にしすぎない。

- 私が話している時のリズム、温度感、言い切りの強さをなるべく残す。
- 意味・意図・情報量を勝手に変えない。

- AIっぽく整った文章、ビジネス文書っぽく安定しすぎた文章にはしない。
- 出力は完成文だけにする。説明、前置き、補足、まとめは不要。

# 残すべきもの
- 熱量のある表現
- 強い言い切り
- 話しながら考えている感じ
- 多少の口語感
- 私らしい言い回し
- 主張の勢い
- ニュアンスのある言い直し
- 「これはこうでした」「ちょっと違っていて」など、思考の修正過程に意味がある表現
- 同じ内容でも、熱量や強調のために繰り返している表現

# 削る・直すべきもの
- 「えー」「あの」「えっと」「その」「なんか」など、意味のないフィラー
- 明らかな音声認識ミス
- 明らかな言い間違い
- 意味のない重複
- 文として読みにくすぎる途切れ
- 不自然な助詞
- 誤変換
- 句読点不足
- 文脈上明らかに不要な脱線

# 言い直しの扱い
- 明確に言い直している場合は、最終的に言いたかった内容が伝わるように整理する。
- ただし、言い直しそのものに熱量やニュアンスがある場合は、完全に消さずに自然に残す。
- 「違う」「そうじゃなくて」「正確には」「というより」などの表現は、思考の流れとして意味がある場合は残す。
- 単なるミス修正だけなら、修正後の内容に自然に統合する。

# 文体
- 原則として、元の話し方のトーンを維持する。
- カジュアルに話している場合は、カジュアルさを残す。
- 丁寧に話している場合は、丁寧さを残す。
- 勝手に「です・ます調」に統一しすぎない。
- 勝手に硬いビジネス文書にしない。
- 勝手に論文調・説明文調にしない。

# 文章の整形
- 句読点を自然に入れる。
- 長すぎる文は、熱量を壊さない範囲で分ける。
- 段落は必要な時だけ分ける。
- 話の勢いがある場合は、短く切りすぎない。
- 読みやすくはするが、整いすぎてつまらない文章にはしない。

# 構造化
- 私が「箇条書きで」「整理して」「見出しをつけて」と言った場合だけ、積極的に構造化する。
- 指示がない場合は、話し言葉の流れを優先する。
- 勝手に要約メモのようにしない。
- 勝手に結論・理由・補足のような型にはめない。

- 固有名詞、数字、URL、メールアドレス、コード、コマンド、ファイル名は勝手に改変しない。

## 5つの結論

1. **CodexはチャットAIではなく「仕事の置き場所」になった** — OpenAI公式PDFが"仕事の文脈をどこへ置くか"という新しい問いを提示。Durable Threadsがその実装形態だ。
2. **Thread automationsで「準備して待つ」スタイルが主流になる** — 30分ごとのSlack監視・週次サマリー生成など、ハートビート型の自律実行がCodexの本来の使い方として公式化された。
3. **「ループを設計する人が勝つ」という認識がエコシステムで広まった** — Peter Steinberger（220万ビュー）・AnatoliKopadze（182万ビュー）・@taiyo_ai_gakuseが独立して同じ結論に達した。ループエンジニアリングが次の競技場だ。
4. **人間の役割は「承認と不可逆な判断だけ」に集約されている** — 探す・読む・整理する・下書きするはCodexの仕事。人間が担うのは送信・公開・責任を持つという不可逆アクションのみ、と公式PDFが明示している。
5. **ナレッジワーカーがユーザーの20%を占め、コーダーより3倍速く増加中** — 500万超の週間アクティブユーザー。Codexの製品アイデンティティが「コーディングツール」から「知的仕事の代行者」に移行した。

## 第1章 「仕事の置き場所」というパラダイムシフト

OpenAI Codex担当者のJason Liuが著した公式PDF「Codex-maxxing for long-running work」（2026年公開、27ページ・10章）は、Codexの使い方についての根本的な問い直しから始まる。

✅（出典：OpenAI公式PDF）
- 「最初に考えるべきは"どんなプロンプトを書くか"ではなく**"この仕事の文脈をどこへ置くか"**」
- 単発プロンプトでCodexを使う場合、毎回同じコンテキストを渡し直す必要がある
- 長く続く仕事には「専用スレッド」が必要だ、という考え方をCodex-maxxingと定義している

---

### Durable Threads——コンテキストを失わずに再開する

10機能の中で最初に紹介されるのがDurable Threads（永続スレッド）だ。これは単なる「会話履歴の保存」ではない。

✅（出典：OpenAI公式PDF）
- **仕事専用の継続スレッド**として機能する。「新しいチャット」を開くたびにコンテキストが消えない
- プロジェクトごと・顧客ごと・ワークストリームごとに専用スレッドを作ることが推奨される
- スレッドはそのまま**Thread automationsの実行基盤**にもなる
- 公式推奨ユースケース：ソーシャルフィードバック監視・OSSプロジェクト・CLI管理・首席補佐官

> "Use a durable thread for work that you expect to revisit."（公式PDF）

---

### 最初に作る3ファイル

Durable Threadsを開くだけでは不十分で、スレッドに「置く情報」を設計することが重要だ。

✅（出典：OpenAI公式PDF）
- **`AGENTS.md`** — Codexの動き方を定義。「承認なしに送信するな」などのルールを記述
- **`workstream.md`** — 現在のタスク・優先順位・進行状況のサマリー
- **`done-definition.md`** — 完了条件の定義。「いつ終わったとみなすか」を事前に書く

この3ファイルをスレッド開始時にCodexに渡すことで、再開のたびにゼロから説明する必要がなくなる。

### ② Voice input——スピードではなく「質の変化」

✅（出典：OpenAI公式PDF）
- 音声入力の価値は「タイピングより速い」ではなく、**整理されていない思考をそのまま渡せる**ことにある
- タイプするとき、人は無意識に情報を削る。音声では「半分しか思い出せない名前」「漠然とした方向性」がそのまま出てくる

> "The benefit is not only speed. It's that spoken input often includes the unedited version of the work."（公式PDF）
>
> サンプルプロンプト例：「Slackで誰かBenっていう人が何か言ってたと思うんだけど、よく覚えてない。とにかく探して。」

---

### ③ Steering——作業中に方向修正する

✅（出典：OpenAI公式PDF）
- Codexが作業中に**次の指示・修正・方向転換を追加**できる機能
- 作業を止めずに「こっちより先にあっちをやって」「この方針は変更する」と割り込める
- 公式5例：Make smaller / Copy wrong / Open PR / Wait deployment / Show preview before posting

### ④ Memory / vault——会話の外に記憶を置く

✅（出典：OpenAI公式PDF）
- **メモリは会話ウィンドウの外に置く**——会話が終わっても記憶が消えない設計
- `vault/` ディレクトリに保存：People / Decisions / Open loops / Daily notes / Project state

> "Repositories hold code. The vault holds rolling context around the work."（公式PDF）

---

### ⑤ Computer & Browser use——3種の使い分け

✅（出典：OpenAI公式PDF）

| 識別子 | 用途 | 特徴 |
|--------|------|------|
| `$browser` | ローカルブラウザ | 認証不要のパブリックサイト向け |
| `@chrome` | 認証済みChromeセッション | ログイン済みサービスへのアクセス |
| `@computer` | GUIアプリ全般 | デスクトップアプリの操作 |

Connectors（Slack・Gmail・GitHub等と直接接続）とSkills（繰り返しワークフローの保存・再利用）も利用可能。

### ⑥ Remote control——別デバイスから承認する

✅（出典：OpenAI公式PDF）
- PCから開始した長時間タスクをスマートフォンから**確認・承認・方向修正**できる機能
- Codexが「人間の承認が必要な判断」に達したときに通知を送る

> "Remote control is not a reason to skip reviews. It is a way to keep enough attention on the loop to unblock the next move."（公式PDF）

Thread automationsはこの公式発表の中でも実用的な応用範囲が最も広い機能だ。

✅（出典：OpenAI公式PDF）
- スレッドに**定期実行のスケジュール**を設定できる機能
- サンプル：「30分ごとにSlack・Gmailをチェックして、返答が必要なメッセージを調査・草案を作成。ただし承認なしに送信しない」
- 問題発生時にSlackへ転送する機能も組み込める

✅（出典：OpenAI公式PDF）

- いずれも「Codexが準備 → 人間が承認」の構造を持つ

---

### ⑨ Goals——ゴール明文化の重要性

✅（出典：OpenAI公式PDF）
- ゴールの書き方で成果が大きく変わる。弱いゴールはドリフトを生む
- 弱い例：「Implement the plan in this Markdown file.」（何のファイルか・いつ終わりか不明）
- 強い例（Rich→Rust移植）："Port this library, keep the public API compatible, and use the original unit tests as the success check."

---

### ⑩ Side panel——スレッド内でレビューする

✅（出典：OpenAI公式PDF）
- Markdown・CSV・スプレッドシート・スライド・Webページをスレッド内でレビューできる
- **内ループ（マイクロ）**：Codexが自律的に試行・修正を繰り返す高速サイクル
- **外ループ（マクロ）**：人間が承認・方向修正する低頻度サイクル。サイドパネルで外ループを回す

## 第3章 3つのループ例とGoals設計

公式PDFのChapter 8「Three loops」は、Thread automationsを使った実践的なループ設計の具体例を3つ示している。いずれも「Codexが準備→人間が承認」の構造で、不可逆アクション前に必ず人間が介在する。

---

### Chief of Staff ループ

✅（出典：OpenAI公式PDF）
- Codexが「首席補佐官」として機能するループ
- 毎朝：受信トレイ確認 → タスクの優先順位付け → 一日の計画を提示する
- 毎週：プロジェクト状況の集約 → 週次レポート生成 → 翌週の準備
- 人間は優先順位の確認と最終承認だけを担当する

---

### フィードバック監視ループ

✅（出典：OpenAI公式PDF）
- 製品・サービスへのフィードバックを常時監視するループ
- Slack・Twitter・サポートチャネル等を30分ごとにチェック
- 重要なフィードバックを検出 → 分類 → 担当者への通知 → 週次サマリー生成

---

### 返金対応ループ

✅（出典：OpenAI公式PDF）
- カスタマーサポートの一形態として返金リクエストを処理するループ
- 受信 → 情報整理 → 対応草案生成 → **人間に送信前確認を求める**
- 「送信する」という不可逆アクションの直前で必ず人間が介在する設計

---

### 強いゴールの4要素

Goals（Chapter 9）はループ設計において最も重要な章だ。

✅（出典：OpenAI公式PDF）

| 要素 | 内容 | 公式例 |
|------|------|------|
| ① 完了条件 | 「〇〇の状態になったら終わり」を明示 | "ready for review when the same tests pass" |
| ② 制約 | 「〜はしない」「〜の範囲内でやる」 | "keep the public API compatible" |
| ③ 検証基準 | 成果物が正しいかをどう判定するか | "use the original unit tests as the success check" |
| ④ エスカレーション | 人間に判断を求めるタイミング | "show me the link before anything is posted" |

## 第4章 Loop Engineeringという概念の出現

公式発表と並行して、「ループを設計する」という概念がエコシステムで独立して盛り上がりを見せている。

---

### Peter Steinberger「プロンプトではなくループを設計しろ」

⚠️（出典：@taiyo_ai_gakuse の分析）
- Peter Steinbergerの投稿「エージェントにプロンプトを打つな。ループを設計しろ」が220万ビューを記録
- 「繰り返しを前提とした仕組みを作る」ことを主張

🔶（出典：@AnatoliKopadze）
- 3エージェント（計画担当・構築担当・判定担当）でゼロからフルアプリを40分で構築するデモを公開
- 「勝者は最も賢いモデルを持っている人ではなく、**最良のループを持っている人**」という発言が拡散

---

### ループ成立の4条件

⚠️（出典：@taiyo_ai_gakuse の分析）

| 条件 | 内容 |
|------|------|
| ① タスクが繰り返す | 週次・日次・常時で発生し続けるタスクであること |
| ② 検証が自動化できる | 成果物の正誤をコードまたはCodex自身が判定できること |
| ③ トークン予算に余裕がある | 長時間実行に必要なコンテキスト量をまかなえるコスト設計であること |
| ④ 適切な道具がある | ループに必要なツール（ブラウザ・API・ファイル等）がすべてアクセス可能であること |

⚠️ 注意：「ループエンジニアリング」はAnthropicが2024年12月に文書化済みのevaluator-optimizerパターンの延長線上にある概念であり、完全に新しいアイデアではない。

### 500万ユーザーと知識労働者への普及

✅（出典：OpenAI公式ホワイトペーパー「The Next Era of Knowledge Work」2026年6月）
- **週間アクティブユーザー：500万人超**（デスクトップアプリリリース以来6倍以上に増加）

### Codexがやること / 人間がやること

✅（出典：OpenAI公式PDF）

**Codexがやること（自律実行）**：探す・読む・整理する・下書きする・選択肢を作る

**人間がやること（不可逆な判断）**：判断する・承認する・**送信する**・**公開する**・責任を持つ

この役割分担は`done-definition.md`に明示的に書いておくことが推奨されている。
