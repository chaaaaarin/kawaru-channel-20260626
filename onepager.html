「仕事の置き場所」というパラダイムシフト

OpenAI公式PDF「Codex-maxxing for long-running work」（2026年6月）は、Codexの使い方についての根本的な問い直しから始まる。✅

「最初に考えるべきは"どんなプロンプトを書くか"ではなく"この仕事の文脈をどこへ置くか"」
今の使い方
  • 毎回チャットを開き直す
  • コンテキストが毎回リセット
  • 「一問一答」しかできていない
  • 自分が離席したら止まる
→
Codex long-runningの姿
  • スレッドをピン留めして継続
  • vaultがプロジェクト文脈を保持
  • 30分ごとにSlackを自動チェック
  • スマホから確認・承認だけすればOK

最初に作る3ファイル（vault最小構成）✅

01

AGENTS.md

Codexの動き方を定義。「承認なしに送信するな」などのルールを記述する

02

workstream.md

現在のタスクと優先順位。Codexが毎回最初に読む地図

03

done-definition.md

完了条件と「人間だけが判断する」ことの一覧

3つのループ例とGoals設計

公式PDF Chapter 8「Three loops」——長時間タスク設計の教科書。いずれも「Codexが準備→人間が承認」の構造。✅

① Chief of Staff
  • 毎朝：受信トレイ確認・計画提示
  • 毎週：週次レポート生成
人間：優先順位・最終承認
② フィードバック監視
  • Slack等を30分ごと監視・分類
  • 担当者通知・週次サマリー
人間：クリエイティブ判断
③ 返金対応
  • 情報整理・対応草案生成
  • 送信前に必ず確認依頼
人間：不可逆アクションの承認

強いゴールの4要素 ✅

要素内容公式例（Rich→Rust移植）
① 完了条件「〇〇の状態になったら終わり」"ready for review when the same tests pass"
② 制約「〜はしない」という境界線"keep the public API compatible"
③ 検証基準成果物の正誤をどう判定するか"use the original unit tests as the success check"
④ エスカレーション人間に判断を求めるタイミング"show me the link before anything is posted"

今すぐ使えるプロンプト

Durable threads 開始プロンプト

このスレッドを[プロジェクト名]専用のDurable threadとして使います。 以下のファイルを最初に読み込んでください。 - workstream.md — 現在のタスクと優先順位 - memory/project-state.md — 背景・決定済み事項・避けること - done-definition.md — 完了条件と人間の承認が必要な判断の一覧

Goals 強いゴール定義テンプレート

ゴール：[何をするか、1文で] ## 完了条件 [「〇〇の状態になったら終わり」を具体的に書く] ## 制約 - [〜はしない] / [使っていいツール・使ってはいけないツール] ## 検証基準 [成果物が正しいかどうかをどう判定するか] ## エスカレーション条件（実行前に確認） - メール・メッセージの送信 / 外部サービスへの公開 / 不可逆な操作
